# 내손안의 가이드 — 안드로이드 앱 비공개 테스트 문제점 및 해결 방안

- 작성일: 2026년 3월 4일
- 대상 버전: Google Play 비공개 베타 (versionCode 3)
- 작성 목적: 비공개 테스트 중 발견된 주요 문제 6가지의 원인과 해결 방향 정리

---

## 앱 전체 구조 (비개발자용 설명)

이 앱은 두 층으로 이루어져 있습니다.

```
[바깥층: 안드로이드 앱 껍데기]
  - Google Play에 올라간 설치 파일 (.aab)
  - 카메라·마이크·위치 같은 하드웨어 권한을 안드로이드에 요청
  - 내부에 브라우저 창(WebView)을 하나 띄움

[안층: 웹 앱 (실제 기능이 모두 여기 있음)]
  - 서버에서 실시간으로 내려받아 실행
  - 화면 전환·AI 가이드 생성·TTS 음성 재생 모두 여기서 처리
  - 가이드북 공유 페이지도 여기서 생성
```

**화면 종류 (앱 안에서 전환되는 화면들):**

| 화면 이름 | 사용자가 보는 것 |
|----------|----------------|
| 시작 화면 (featuresPage) | 앱 처음 켰을 때 나오는 온보딩/랜딩 화면 |
| 메인 화면 (mainPage) | 카메라·사진 업로드·음성 입력 화면 (핵심 기능) |
| 결과 화면 (detailPage) | AI가 생성한 가이드 텍스트 + 음성 재생 화면 |
| 보관함 (archivePage) | 저장된 가이드 목록 |
| 설정 (settingsPage) | 언어 변경, 알림 등 설정 |
| 추천갤러리 공유 페이지 | 다른 사람이 만든 가이드북을 보는 별도 화면 |

## 문제점 1: 앱을 껐다 켜거나 언어를 바꾸면 로그인이 풀림

### 현상
사용자가 구글/카카오로 로그인했는데, 앱을 껐다 켜거나 언어를 바꾸면 다시 로그인 화면이 나타남.

### 원인
> 비유: 마트 직원이 어제 찍어둔 오래된 사진으로 오늘 손님을 안내하는 상황

앱 안의 웹 부분은 인터넷에서 파일을 받아 실행됩니다. 그런데 앱에 **"오래된 파일 저장고"(서비스워커 캐시)**가 있어서, 개발팀이 서버에 새 파일을 올려도 이 저장고에서 구버전 파일을 계속 꺼내 씁니다.

결과적으로:
- 서버에는 최신 로그인 코드가 있어도
- 앱은 저장고의 구버전 코드를 실행
- 구버전 코드가 로그인 정보를 제대로 읽지 못해 로그인이 풀린 것처럼 보임

언어 변경 시: 화면을 새로 불러오는 순간 같은 현상 발생.

(참고: 로그인 기록 자체는 DB에 7일간 안전하게 저장됨. 서버를 재시작해도 로그인 기록은 사라지지 않음.)

### 해결 방향
저장고 규칙을 바꿉니다.

- **현재 규칙:** 저장고에 있으면 저장고 것을 먼저 씀 (구버전 계속 사용)
- **바꿀 규칙:** 항상 인터넷(서버)에서 최신 파일을 받아 씀. 인터넷이 안 되는 경우에만 저장고 파일 사용

오프라인에서도 볼 수 있어야 하는 **공유 가이드북 페이지**는 기존 방식(저장고 우선) 그대로 유지.

### 수정 후 예상 시나리오
- 앱을 껐다 켜도: 항상 최신 파일 실행 → 로그인 상태 유지
- 언어 변경 후: 최신 파일 실행 → 로그인 상태 유지
- 오프라인 상태에서 저장한 공유 가이드북: 변경 없이 정상 작동

## 문제점 2: 카메라·마이크·위치 권한을 앱 켤 때마다 다시 물어봄

### 현상
처음 앱 설치 후 카메라·마이크·위치 권한을 허용했는데, 앱을 끄고 다시 켜면 또 허용 팝업이 뜸.

### 원인
> 비유: 매장 입구에서 신분증을 확인하는 직원이, 한 번 확인했다는 사실을 메모하지 않아 같은 손님에게 매번 신분증을 요구하는 상황

안드로이드 앱 껍데기에 "권한을 한 번 허용받으면 기억해라"는 코드가 없습니다. 그래서 WebView(앱 안의 브라우저)가 카메라·마이크 등을 요청할 때마다 안드로이드 시스템이 새로 팝업을 띄웁니다.

### 해결 방향
앱 껍데기 코드에 "권한을 허용받으면 기억해서 다음엔 묻지 말라"는 처리를 추가합니다. 또한 앱 설정 파일에 사용하는 권한 목록(카메라·마이크·위치)을 명시합니다.

**중요:** 이 수정은 앱 껍데기(설치 파일) 변경이므로, **Google Play에 새 버전(versionCode 4)을 다시 올려야 합니다.**
이번 작업에서는 사용자 승인 후 별도로 진행 예정.

### 수정 후 예상 시나리오
- 앱 최초 설치 후: 권한 팝업 1회 표시
- 허용 후 앱 재시작: 팝업 없이 바로 사용 가능
- 앱 삭제 후 재설치: 1회만 다시 요청

## 문제점 3: 추천갤러리 공유 페이지에서 뒤로가기 버튼을 누르면 보관함이 아닌 엉뚱한 화면이 나옴

### 현상
보관함에서 추천 가이드를 열어 보고 왼쪽 상단 ← 버튼을 눌렀을 때, 보관함이 아닌 메인 화면이나 로그인 화면으로 이동됨.

### 원인
> 비유: 도서관에서 책을 읽다가 출구 표지판을 따라갔더니 주차장이 아닌 옆 건물로 나오는 상황

추천갤러리 공유 페이지의 뒤로가기 버튼이 "이전 화면으로 돌아가라"는 명령 대신, **"보관함 주소로 새로 이동하라"**는 명령을 수행합니다.

안드로이드 WebView 안에서 "새 주소로 이동"을 하면:
1. 앱 내부 화면 히스토리가 초기화됨
2. 새로 주소로 이동하면서 웹 파일을 다시 불러옴
3. 구버전 저장 파일이 로드되면서 로그인 정보까지 영향받을 수 있음

일반 공유 페이지의 X(닫기) 버튼도 같은 계열 문제: "창 닫기" 명령이 WebView 안에서는 아무 동작도 안 함.

### 해결 방향
뒤로가기 버튼 명령을 바꿉니다:
- 현재: "보관함 주소로 새로 이동"
- 변경: "방금 왔던 화면으로 돌아가기"

X 버튼도: "창 닫기 안 되면 이전 화면으로 돌아가기" 대안 동작 추가.

### 수정 후 예상 시나리오
- 보관함 → 추천갤러리 → ← 버튼: 보관함으로 정확히 복귀
- 메인화면 → 추천갤러리 → ← 버튼: 메인화면으로 복귀
- 로그인 상태 변화 없음 (화면을 새로 불러오지 않으므로)

## 문제점 4: 구글/카카오 로그인 완료 후 보관함으로 이동됨

### 현상
앱에서 구글 또는 카카오 로그인을 하면, 완료 후 카메라·촬영 화면(메인 기능 화면)이 아닌 보관함 화면으로 이동됨.

### 원인
> 비유: 호텔 체크인을 마쳤는데 직원이 고객을 객실이 아닌 수영장으로 안내하는 상황

안드로이드 앱에서는 로그인 창을 새 팝업으로 열지 못합니다(안드로이드 보안 정책). 대신 현재 화면에서 구글/카카오 로그인 페이지로 이동했다가 돌아옵니다.

로그인이 완료되면 서버가 "어디로 보낼까?" 를 결정하는데, 현재 코드에 **"보관함으로 이동"** 이 적혀 있습니다. 사용자가 원하는 화면은 **카메라·음성 입력 화면(메인 화면)**입니다.

### 해결 방향
로그인 완료 후 이동 주소를 바꿉니다:
- 현재: 보관함으로 이동
- 변경: 카메라·촬영·음성 입력 화면으로 이동

구글 로그인과 카카오 로그인 두 곳 모두 같은 수정 적용.

### 수정 후 예상 시나리오
- 구글 로그인 완료: 카메라·사진·음성 입력 화면(메인 화면) 표시
- 카카오 로그인 완료: 동일
- 이미 로그인된 상태에서 앱 재시작: 기존과 동일 (변경 없음)

## 문제점 5: 음성 읽기(TTS) 버튼이 간헐적으로 눌린 채 멈추거나 반응이 없음

### 현상
AI 가이드 결과 화면에서 음성 재생 버튼을 눌렀을 때:
- 버튼이 "정지" 아이콘으로 바뀌는데 실제 음성은 안 나옴
- 또는 버튼이 "재생" 아이콘인데 왜 안 나오는지 모르는 상태
- 기기마다 증상이 다름

### 원인
> 비유: 식당에서 주문을 받은 직원이 주방에 전달하기 전에 이미 "주문 완료" 팻말을 올려두는 상황. 주방이 바빠서 요리를 못 만들어도 팻말은 그대로 남아 있음

현재 코드 순서:
1. 버튼을 "정지" 아이콘으로 먼저 바꿈 (시각적 피드백)
2. 그 다음 음성 재생 시작 시도

문제: 안드로이드 Chrome 브라우저는 사용자가 직접 터치하지 않은 상황에서 자동으로 소리를 내는 것을 막는 경우가 있습니다. 이 때:
- 재생이 실패해도 버튼은 이미 "정지" 아이콘인 상태로 굳어버림
- 어떤 기기는 실패 신호(onerror)를 보내고, 어떤 기기는 아무 신호도 안 보냄
- 신호가 없으면 버튼이 영원히 "정지" 아이콘으로 남음

추가로, 음성 목록(voices)이 앱 처음 시작 시 비어있는 경우가 있어 음성 선택 자체가 실패하기도 함.

### 해결 방향
1. **음성 목록 사전 준비:** 앱 시작과 동시에 음성 목록을 미리 불러와 저장해 두기
2. **버튼 상태 순서 변경:** 음성이 실제로 시작됐을 때 버튼 아이콘 변경
3. **안전 타이머 추가:** 버튼을 눌렀는데 1.5초 내에 음성이 시작되지 않으면 자동으로 버튼을 "재생" 아이콘으로 되돌림 (사용자가 다시 누를 수 있게)

### 수정 후 예상 시나리오
- 음성 재생 성공: 버튼이 "정지" 아이콘 → 읽기 완료 후 "재생" 아이콘으로 복귀
- 음성 재생 실패(자동 차단): 1.5초 후 버튼이 "재생" 아이콘으로 복귀 → 사용자가 다시 탭하면 정상 재생
- 기기 무관하게 일관된 동작

## 문제점 6: 보관함·프로필·관리자 화면에서 음성 버튼이 안드로이드 Chrome에서 작동 안 함

### 현상
아래 화면들에서 음성 재생 버튼을 눌러도 음성이 멈추지 않거나, 멈춘 후 다시 재생이 안 됨:
- 보관함에서 저장된 가이드 열어볼 때
- 프로필 페이지에서 가이드 미리보기
- 관리자 대시보드에서 가이드 미리보기
- 다른 사람과 공유된 가이드북 페이지

### 원인
> 비유: 음악 플레이어의 일시정지 버튼이 안드로이드 폰에서는 그냥 전원 끄기처럼 동작해서, 다시 재생 버튼을 눌러도 곡이 처음부터 시작되지 않는 상황

안드로이드 Chrome 브라우저에는 **알려진 버그(Chromium 버그 #679437)**가 있습니다:
- 일반 브라우저(PC Chrome, iOS Safari): "일시정지" 명령이 정상 동작
- 안드로이드 Chrome: "일시정지" 명령이 음성을 완전히 꺼버리고 재생 불가 상태로 만듦

현재 4개 화면의 음성 중지 코드가 이 방식(일시정지 후 취소)을 사용하고 있습니다:
- 보관함 가이드 열람 화면
- 프로필 페이지 미리보기
- 관리자 대시보드 미리보기
- 공유 가이드북 페이지

(참고: 메인 AI 결과 화면의 공유 페이지는 이미 수정 완료됨)

### 해결 방향
4개 화면 모두에 동일한 수정을 적용합니다:
- **안드로이드 Chrome인지 자동 감지**
- 안드로이드 Chrome이면: 일시정지 건너뛰고 바로 중단 명령만 실행
- 그 외 기기(iOS, PC 등)는: 기존 방식 유지

### 수정 후 예상 시나리오
- 안드로이드 Chrome: 음성 재생 중 버튼 클릭 → 즉시 멈춤 → 다시 클릭 시 재생
- iOS, PC Chrome: 기존과 동일하게 작동
- 4개 화면 모두 동일한 동작 보장

## 수정 순서 계획

코드 수정은 아래 순서로 진행 예정 (각 항목은 서로 독립적):

1. 캐시 저장고 규칙 변경 (#1) — 즉시 효과, 전체 문제에 기반 영향
2. 로그인 후 이동 주소 수정 (#4) — 간단한 주소 변경
