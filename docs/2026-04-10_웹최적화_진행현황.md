## 2026-06-30 삼성 TTS 버그 수정 — 같은문단 반복 + 첫무음 + 일시정지 먹통 (코드 수정 적용)

### 📋 사용자 보고 증상 (삼성폰만, iOS 정상)
- 삼성폰(웹/TWA=순수 Chrome 엔진)에서 음성 자동재생이: (1) 첫 접속 시 무음, 재시도하면 됨 (2) 같은 문단을 반복 재생
- 텍스트는 이미 전체 생성 완료 상태 (온디바이스 TTS라 레이턴시 불필요한데 타임아웃 로직 있음)
- 사용자 통찰: "재생 타임 설정(워치독) 로직이 의심된다, 웹 버전 버그로 보임" → **정확했음**

### 🔍 진단 (운영본 Chrome DevTools 직접 검증 + 4회 딥리서치 + 적대적 검증)
- `isNativeApp = !!window.ReactNativeWebView` → TWA는 false → 웹 `synth.speak()` + 워치독 경로. iOS 앱은 네이티브 TTS(자체 완료신호)라 무관.
- iOS가 정상인 이유: 위험 로직(`synth.cancel()`, 워치독)이 전부 `detectPlatform() !== 'ios'` 가드로 꺼져 있음. 삼성/안드로이드만 노출.
- 운영본 배포 코드([assets/main 번들](https://my-handyguide1.replit.app/assets/) Chrome DevTools 검증)가 정적 분석과 100% 일치 확인.

### ✅ 확정된 근본 원인 (코드 줄번호)
1. **같은 문단 반복**: 워치독 `setTimeout`([index.js](public/index.js) speakNext 내)과 실제 `onend`가 **둘 다** `speakNext()` 호출 → 워치독이 먼저 터진 뒤 늦은 onend가 또 호출 = 이중 진행. 셋(onend/onerror/워치독)이 "이미 진행했는지" 확인하는 공용 가드 없음.
2. **워치독 조기 발동**: `estimatedDuration` 계산식이 `utterance.rate = 0.9`(10% 느리게)를 미반영 → 실제보다 짧게 잡혀 재생 끝나기 전 발동.
3. **첫 무음**: 잠금해제(initSpeechUnlock)가 `getVoices()`/`voiceschanged`를 안 기다림 → 삼성 음성 목록 지연 로딩으로 첫 재생 `voice=null` 무음.
4. **(검증 중 추가 발견) 일시정지 먹통**: 워치독 핸들이 클로저 안에만 있어, 일시정지 중 좀비 워치독이 발동해 `synth.cancel()`로 멈춘 음성 파괴 → 재개(▶) 먹통 (11초+ 일시정지 시 결정론적).

### 🔬 워치독 "삭제" vs "유지" 검증 (사용자: "iOS처럼 꺼버리면?")
- 적대적 검증 결과 **완전 삭제는 위험**: 안드로이드는 onend가 영영 안 오는 경우 현재도 실재(Chromium #679437 ~15초 컷오프 onend/onerror 없이 멈춤, IBM 2024-08 모바일 start-only). 삭제 시 음성 영구 멈춤.
- iOS가 워치독 없이 되는 건 ① 네이티브 TTS의 speakDone 자체신호 ② 로컬음성이라 onend 안정 — 안드로이드(웹 synth + remote 음성)와 다른 엔진.
- **결론: 삭제 폐기, 워치독 유지하되 조기발동만 교정 + 이중진행 빗장.**

### 🔧 적용한 수정 (public/index.js 한 파일, +34/-4줄)
1. **1회용 빗장**: utterance 생성 직후 `let advanced = false`(클로저 지역변수) + onend/onerror/워치독 세 콜백 **최상단**에 `if (advanced) return; advanced = true;` → 먼저 도착한 1개만 진행.
2. **워치독 시간 교정**: `estimatedDuration = Math.max((len / 4 / 0.9) * 1000, 3000) + 8000` (rate 0.9 반영, 박제 없이 완전 교체).
3. **첫 무음**: speakNext 맨 앞 `!__ttsVoicesReady && !isNativeApp && detectPlatform() !== 'ios'` 조건에서만 최초 1회 `getVoices()` 최대 ~1초 비동기 폴링.
4. **좀비 워치독 취소**: `window._currentWatchdog`에 핸들 저장 + resetSpeechState·일시정지(non-native)에서 `clearTimeout`.

## 2026-06-30 이미지 업로드 "구글 포토 강제" 문제 — 진단 완료(OS 정책, 코드 수정 불가 결론)

### 📋 사용자 보고 증상 (실기기)
- 기기: **Galaxy A36 (SM-A366U), One UI 8.0 / Android 16**, 빌드 A366USQ4BYK3
- 증상: 이미지 업로드 버튼 클릭 → **삼성 갤러리(내 보관함)가 아니라 구글 포토(클라우드)가 기본으로 뜸**. 이전엔 바로 여러 옵션(갤러리/기기 내)이 떴음.
- 구글 포토 기본 화면의 **클라우드 사진**을 고르면 다운로드 때문에 느림/멈춤("타임아웃 느낌"). 로컬 사진은 **"컬렉션 > 기기 내" 2단계**를 거쳐야 보이고, 그건 정상 업로드됨.
- **iOS(아이폰12)는 업로드 정상** — 무관. (사파리 TTS 음성 미출력은 업로드와 무관한 별개 이슈: iOS 사파리 사용자 제스처 차단, 앱은 WebView가 완화해 정상.)
- 핵심 전제: 마지막 코드 변경 2026-05-11 이후 **앱 코드/설정 변경 0건** → 원인은 앱이 아니라 외부(OS) 변화.

### 🔍 진단 과정 (딥리서치 다회 + 적대적 검증)
초기 추측 가설들은 실증상과 달라 **순차적으로 정정**됨 (추측 기반 진단의 위험 사례):
- ❌ "검은화면/FileReader 실패" → 실증상은 "구글 포토가 열림"(다른 증상)
- ❌ "iOS 카메라 회귀" → iOS 업로드는 정상
- ❌ "저장소 권한(READ_MEDIA_IMAGES) 부재" → `<input type=file>`은 권한 불필요(공식 문서, 기각)

### ✅ 확정된 근본 원인 (Chromium 소스 verbatim 검증)
- **One UI 8 / Android 16이 구글 Photo Picker를 강제 기본화.** 코드의 `accept="image/*"`([public/index.html:768](public/index.html#L768))는 Chromium 소스(`SelectFileDialog.java`의 `shouldUsePhotoPicker`/`isSupportedPhotoPickerTypes`)상 **MIME 앞부분이 `image/`로 시작하면 무조건 구글 Photo Picker를 띄우는** 정확한 조건. png/jpeg 구체 지정도 동일(소스가 `startsWith("image/")`만 검사).
- **AOS 운영본은 react-native-webview가 아니라 TWA**(순수 Chrome 엔진). 그래서 "웹도 동일" 증상이 당연하고, 네이티브 우회(onShowFileChooser 등)는 적용 불가.

### 🧪 시도 → 적대적 검증 → 롤백 (octet-stream)
- **커밋 전 시도**: `accept="image/*"` → `accept="image/*,application/octet-stream"` ([public/index.html:768](public/index.html#L768)) 한 줄 변경 (구글 Photo Picker 우회 목적).
- **적대적 재검증 결과 반증**(제11조 미달):
  - octet-stream 등 비미디어 MIME을 섞으면 갤러리가 아니라 **DocumentsUI(파일 탐색기)로 빠짐** → 오히려 구글 드라이브/클라우드 노출 증가 가능 (Chromium issue 428394446, AOSP MimeFilterUtils).
  - WebView에선 우회가 깨진다는 실측(Home Assistant android #6055, S25U/One UI 8/A16).
  - One UI 8/A16/A36에서 octet-stream이 실제로 통한다는 **긍정 실측 0건**.
- **즉시 롤백 완료** — `git diff` 비어있음(원본 `accept="image/*"`와 100% 동일).

### 🎯 최종 결론 — 앱 코드로 해결 불가 (OS 정책)
- HTML/accept로 "로컬 사진 먼저 띄우기" **불가능**. accept는 (구글 Photo Picker ↔ DocumentsUI 파일탐색기) 이분법만 제어. "갤러리 그리드 유지 + Photo Picker 회피"는 양립 불가.
- 네이티브 Photo Picker API에도 "기본 탭/클라우드 비활성화" 파라미터 자체가 없음. 앱 제어 불가 — **시스템/OEM/사용자만** 통제.
- **이건 우리 앱 버그가 아니라 Android 16 / One UI 8이 바꾼 OS 동작.** 코드로 막으려는 시도는 (octet-stream처럼) 멀쩡한 동작을 깨뜨릴 위험만 큼.
